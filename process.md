# 🚀 OSS-Vision ドメイン移管＆インフラ再構築 激闘の完全版記録

**構築時期:** 2026年4月
**対象ドメイン:** `oss-vision.com`
**最終インフラ構成:** お名前.com (ドメイン管理/DNS) / Google Workspace (メール/Webサイト) / GitHub Pages (HTTPS転送サーバー)

---

## 第0章：事の発端（Squarespaceからの脱出と真の目的）
物語の始まりは、Google Workspaceの「カスタムドメイン設定（Googleサイトへの独自ドメイン割り当て）」が正常に機能していなかったことに遡る。

当時のドメイン管理は、Google Domainsの事業売却に伴い移管先となっていた「Squarespace」にあった。
「カスタムドメインの不調を根本から見直し、より管理しやすい環境でインフラを正しく再構築する」という強いモチベーションのもと、日本の老舗レジストラである「お名前.com」へのドメイン移管プロジェクトが幕を開けた。

## 第1章：移管完了と予期せぬ完全沈黙（見えない南京錠との死闘）
お名前.comへの移管手続き自体は完了したものの、直後にWebサイトへのアクセスと `@oss-vision.com` 宛のメール送受信が「完全停止」するという未曾有のインフラ障害が発生する。

各種設定（Aレコード、MX、CNAME）はお名前.com側で完璧に行われていたが、世界中のDNS（`8.8.8.8`など）から `SERVFAIL`（通信拒否）を食らう状態になっていた。

* **原因:** 旧環境（Google Domains/Squarespace時代）で設定されていた「DNSSEC（ドメインの身元を保証する暗号化署名）」の古い鍵（DSレコード）が、上位機関に残留したまま移管されてしまったため。新しいお名前.comの鍵と不一致を起こし、インターネット全体から「偽装ドメイン」と判定され遮断されていた。
* **解決策:** お名前.comのユーザー画面からは削除できない裏側の仕様だったため、サポートへ「移管元でDNSSECが有効なまま移管してしまった。至急DSレコードの強制削除を求む」と的確に依頼。見事、古い南京錠の破壊に成功し、通信レイヤーが開通した。

## 第2章：Google Workspace 鉄壁のキャッシュ
DNSは正常に開通したものの、Webサイトを開くと `ERR_CONNECTION_CLOSED` となり、依然としてアクセスできない状態が続いた。

* **原因:** Google Workspaceの管理コンソールが、DNSSECエラー時代の「無効なDNSです」という状態を強くキャッシュ（記憶）しており、サイトを表示するためのSSL証明書（HTTPSの鍵）の発行処理をストップさせていた。
* **解決策:** 管理コンソールから手動で「詳細」→「再確認」のトリガーを引き、Googleのシステムに強行的に最新のDNS状態をチェックさせた。これによりSSL証明書の発行が再開され、無事に `https://www.oss-vision.com` の表示とメールの送受信が復旧を果たした。当初の目的であった「カスタムドメインの正常化」をここで達成する。

## 第3章：お名前.comからの「時限爆弾」
インフラ復旧の安堵も束の間、お名前.comから「Whois情報の不備により、期日までに修正しないとドメインを利用制限（停止）する」という恐ろしい警告メールが届く。

* **原因:** 移管時に「管理・技術・経理」の各担当者情報が `#temporarily set#` というダミーデータで強制入力されていたため、ICANNの国際ルール違反となっていた。
* **解決策:** お名前.comの複雑怪奇なUIのトラップ（「登録者名義変更」という厳格なメール認証ループにハマる罠）を自力で回避。Whois情報変更画面から「登録者情報には触れず、下部3つの担当者情報だけをコピー上書きする」という最短ルートを見つけ出し、無事に時限爆弾を解除した。

## 第4章：完璧なる「ネイキッドドメイン」転送網の構築（神ハック）
すべての問題が解決したと思われたが、名刺やQRコードで使われる `oss-vision.com`（wwwなし）に HTTPS でアクセスした際、エラー画面になってしまうという現代特有の課題に直面する。

* **原因:** Google Workspaceが提供するネイキッドドメインの転送機能は「HTTP（暗号化なし）」のみのサポートであり、ブラウザが標準で行う HTTPS アクセスを受け止めきれないというGoogle側の仕様的限界。
* **解決策:**
    1. GitHub上に転送専用のリポジトリ（`web`）を作成し、0秒転送を行う `index.html` を配備。
    2. GitHub Pages の機能を利用し、独自ドメイン（`oss-vision.com`）を設定して**無料のSSL証明書を自動発行**させる。
    3. お名前.comのAレコードをGoogleのIPからGitHubのIPへ向け直す。
    4. DNSの浸透（TTLのキャッシュ切れ）を根気強く待ち、最後にGitHub側で「Enforce HTTPS」を有効化。

## 🌟 結末：最強のインフラ完成
度重なる障害とシステムの壁を「自らの手とターミナルでの調査」によって打ち破り、最終的に以下の完璧なルーティングシステムが完成した。

1. **`https://www.oss-vision.com`** 👉 Google Sitesで美しく安全に表示される。（本来の目的達成）
2. **`https://oss-vision.com`** 👉 GitHub PagesがHTTPSアクセスを安全に受け止め、即座に公式URLへ自動転送する。（死角なし）
3. **`@oss-vision.com`** 👉 Google Workspaceで安定してメール送受信が行われる。（完全復旧）

「なんとか動く」レベルではなく、「現代のWebインフラとして最も美しく、強固な状態」へと昇華された。
